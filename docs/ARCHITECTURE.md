# Архитектура

## Назначение

Проект сделан как расширяемый каркас семейного Telegram-бота. Сейчас реализованы доступ к боту, раздел семьи, раздел рецептов и меню недели. Будущие домены покупок должны добавляться рядом с существующими модулями, не раздувая текущие хендлеры.

## Основные модули

`bot.py`

Точка входа. Загружает конфиг, открывает базу, инициализирует схему, создает `Bot` и `Dispatcher`, кладет объект базы в `dispatcher["db"]`, подключает роутеры и запускает polling.

`app/config.py`

Загружает `.env`, валидирует `BOT_TOKEN`, задает путь к базе `data/bot.db` и username бота `Family_Menuu_Bot`.

`app/database/storage.py`

Слой доступа к SQLite. Методы объявлены async, чтобы хендлеры aiogram могли работать с базой единообразно. Внутри используется стандартный `sqlite3`.

Текущие таблицы:

- `users`: пользователи семьи.
- `invitations`: одноразовые приглашения.
- `categories`: категории рецептов.
- `recipes`: рецепты.
- `recipe_ingredients`: ингредиенты рецептов.
- `menus`: недельные меню по дате понедельника.
- `menu_items`: блюда в меню, с днем недели или без дня.

`app/handlers/`

Aiogram-роутеры:

- `start.py` - `/start`, первый владелец, вход по приглашению, отказ без доступа.
- `family.py` - список семьи и генерация приглашений владельцем.
- `recipes.py` - раздел рецептов, FSM добавления и редактирования, просмотр и удаление.
- `weekly_menu.py` - просмотр, добавление и редактирование меню недели.
- `menu.py` - заглушка раздела покупок.
- `common.py` - общий guard доступа и текст отказа.

`app/keyboards/`

Клавиатуры Telegram:

- `main.py` - постоянное нижнее меню.
- `family.py` - inline-кнопка «Пригласить» для владельца.
- `recipes.py` - inline-клавиатуры раздела рецептов.
- `weekly_menu.py` - inline-клавиатуры меню недели.

`app/services/`

Небольшие helpers без привязки к Telegram API. Сейчас здесь генерация ссылки приглашения, парсер ингредиентов, расчёт недель и агрегация ингредиентов меню.

`app/middlewares/`

Aiogram middleware для диагностики. `LoggingMiddleware` пишет все входящие сообщения, callback-нажатия и исключения в общий лог.

`app/logging_config.py`

Настраивает root logger: поток в консоль и rotating file `logs/bot.log`.

## Поток регистрации

1. Пользователь отправляет `/start`.
2. Бот ищет пользователя по `telegram_id`.
3. Если пользователь уже есть, бот показывает главное меню.
4. Если пользователь пришел с аргументом `/start CODE`, бот пытается погасить приглашение.
5. Если приглашение валидно и не использовано, создается `member`.
6. Если приглашения нет или оно невалидно, бот пытается создать первого владельца.
7. Если в базе еще нет пользователей, создается `owner`.
8. Если пользователи уже есть, новый пользователь получает отказ.

## Приглашения

Приглашение создается владельцем в разделе «👨‍👩‍👧 Семья». Код генерируется через `secrets.token_urlsafe(8)`, сохраняется в `invitations`, а ссылка строится как:

```text
https://t.me/Family_Menuu_Bot?start=CODE
```

После первого успешного использования у приглашения заполняются `used_by_user_id` и `used_at`. Повторное использование того же кода возвращает отказ.

## Расширение схемы

Новые таблицы лучше добавлять в `Database.init_schema()` отдельными блоками `CREATE TABLE IF NOT EXISTS`. Текущий домен рецептов уже использует:

- `recipes` - рецепты.
- `recipe_ingredients` - ингредиенты рецептов.

Меню недели использует:

- `weekly_menus` и `menu_items` - меню недели.

Фактическое имя таблицы меню: `menus`.

Для следующих этапов ожидаемые домены:

- `shopping_lists` и `shopping_items` - списки покупок.

Для каждого нового домена стоит держать SQL-методы в `app/database/storage.py` до тех пор, пока файл не станет слишком большим. После этого можно разделить слой базы на несколько репозиториев, но внешний API хендлеров должен остаться простым.

## Текущие ограничения

- Нет миграционного инструмента. На раннем этапе схема создается через `CREATE TABLE IF NOT EXISTS`.
- Нет разделения семей: все пользователи в базе считаются одной семьей.
- Нет фоновых задач и webhook-режима.
- Раздел покупок пока не реализован.
